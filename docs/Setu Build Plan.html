Why this shape Backend and frontend are separate top-level folders because they deploy to two different places (Render for backend, a static host for frontend) and are written in two different languages. Each of the three ML models gets an identical internal structure (collect → clean → features → train → evaluate → predict) so once you understand one, you understand all three.
PHASE 0

Accounts & environment setup

Nothing gets built until these accounts exist and keys are in hand. This is a once-only checklist, not ongoing work.

1

Create the accounts you'll need

  • GitHub — hosts the code, connects to both deploy targets
  • Render — will host the Python backend
  • Supabase — will host the Postgres database
  • Vercel or Netlify — will host the React frontend (pick one; either works fine)
  • Anthropic or OpenAI account — for the LLM calls (alert text generation, reasoning narration)
2

Get free API keys for live data sources

  • OpenAQ — free, no key required for basic use, confirm current requirements
  • OpenWeatherMap — free tier API key
  • AQICN / waqi.info — free API token, request via their site
  • Open-Meteo — free, no key required
Definition of done
  • All five accounts created
  • All API keys/tokens saved somewhere safe (a password manager or private note — never committed to git)

PHASE C

Model 1 — AQI (build the ML pipeline once, here)

BackendMLLearn the pattern here

Your first trained model, and the template every later model reuses. Take this one slowly — once you understand each step here, Cyclone and Flood are the same steps again with different data.

1

Collect historical AQI + weather data

A script that downloads a multi-year historical dataset for the covered cities from CPCB/OpenAQ archives and Open-Meteo's historical weather data, saving raw files locally.

Files created
backend/app/ml/aqi/collect.pybackend/data/raw/aqi_*.csv
2

Clean the data

Handle missing values, fix mismatched units, remove obviously broken readings, align timestamps across the AQI and weather sources.

Files created
backend/app/ml/aqi/clean.pybackend/data/processed/aqi_clean.csv
3

Engineer features

Turn raw cleaned readings into the actual inputs the model learns from: recent AQI history at multiple lags, rate-of-change over recent days, wind speed/direction, humidity, temperature, day-of-year (for seasonal burning patterns).

Files created
backend/app/ml/aqi/features.py
4

Split into train and test sets — by time, not randomly

Older data trains the model, newer data tests it. This step lives inside train.py but is called out on its own because getting it right (never split time-series data randomly) is the most important correctness rule in the whole pipeline.

5

Train the model

Train an XGBoost (or LightGBM) regressor on the engineered features to predict AQI 24-72 hours ahead. Save the trained model to disk.

Files created
backend/app/ml/aqi/train.pybackend/app/ml/aqi/model.pkl
6

Evaluate honestly

Run the trained model against the held-out test data. Compute accuracy/error margin, precision, recall, and false-positive rate on spike detection. Save the results.

Files created
backend/app/ml/aqi/evaluate.pybackend/app/ml/aqi/report.json
7

Build the live inference function

A function that loads the saved model and, given current live readings for a city, returns a prediction plus a confidence score — this is what Stage 1 will call continuously.

Files created
backend/app/ml/aqi/predict.py
8

Expose predictions via API

Files created
backend/app/routers/predictions.py

PHASE E

Model 2 — Cyclone

BackendML

Same nine steps as Phase C, different data and features. Scoped to landfall risk and intensity trend for an already-tracked storm — not full track forecasting.

1–9

Repeat the Phase C pipeline for cyclone data

  • Collect IMD RSMC New Delhi best-track data and/or IBTrACS (NOAA) historical cyclone data
  • Clean and align by storm ID and 6-hourly timestep
  • Engineer features: central pressure, wind speed, pressure drop rate, track position/direction, sea surface temperature, season
  • Time-based train/test split (split by storm/season, not randomly)
  • Train a classification model for landfall probability + regression for intensity trend
  • Evaluate honestly, save report.json
  • Build the live inference function
  • Expose via the same predictions router
  • Curate one real named historical cyclone (e.g. Amphan) for replay
Files created
backend/app/ml/cyclone/*.pybackend/app/ml/cyclone/model.pklbackend/app/ml/cyclone/report.jsonbackend/data/replay_events/cyclone_event.json
Definition of done
  • Same checklist as Phase C, applied to cyclone data
  • Cyclone predictions flow through the exact same Stage 1 pre-staging pipeline built in Phase D, with no changes needed to that pipeline itself
PHASE F

Model 3 — Flood

BackendMLHardest data, build last

Same pipeline again, but budget more time for cleaning — this is the weakest data situation of the three, as flagged in the build plan.

1–9

Repeat the pipeline for flood data

  • Collect a curated rainfall-flood dataset (regional CSVs) as the primary source; layer in India-WRIS river data opportunistically, expecting manual curation work
  • Clean — expect this step to take longer here than for AQI or cyclone
  • Engineer features: rainfall accumulation over recent days, regional historical flood-proneness, available discharge data
  • Time-based train/test split
  • Train a classification model for flood risk tier
  • Evaluate honestly — expect coarser numbers than the other two models, and say so in the report
  • Build live inference function, expose via API
  • Curate one real historical flood event for replay
Files created
backend/app/ml/flood/*.pybackend/app/ml/flood/model.pklbackend/app/ml/flood/report.jsonbackend/data/replay_events/flood_event.json
Definition of done
  • Same checklist as Phase C/E, applied to flood data
  • report.json for flood is honest about lower confidence/precision than AQI or cyclone, if that's what the numbers show — don't inflate this
